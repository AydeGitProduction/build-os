## Why DB Tables Are Not Yet Created

The deployed migration route (`/api/admin/migrate-p11-5`) uses `pg.Client` with the Supabase Pooler. This returns "Tenant or user not found" — a consistent error across all admin migration routes in this deployment (including the existing `migrate-p9b`).

**Root cause:** The Supabase Supavisor (pooler) requires the actual PostgreSQL database password, not the JWT service_role key. The codebase uses `SUPABASE_SERVICE_ROLE_KEY` (JWT) as the pg password — which Supabase stopped accepting in 2024+.

**Evidence this is not new:** The `migrate-p9b` route has the same failure mode. The tables it was supposed to create (wizard_conversations, wizard_assumptions, wizard_state) exist because they were run manually in the Supabase SQL Editor at some point.
